## 通用说明

### 认证头
所有需要认证的接口都需要在请求头中携带 JWT Token：
```
Authorization: Bearer <token>
```

### 响应格式
成功响应：
```json
{
  "message": "操作成功",
  "data": {...}
}
```

错误响应：
```json
{
  "code": 404,
  "error": "NOT_FOUND",
  "message": "资源不存在",
  "detail": "详细说明（可选）"
}
```

### HTTP状态码
| 状态码 | 描述 |
|--------|------|
| 200 | 请求成功 |
| 400 | 请求参数错误 |
| 401 | 未认证或Token无效 |
| 403 | 无权限访问 |
| 404 | 资源不存在 |
| 409 | 资源冲突 |
| 429 | 请求过于频繁 |
| 500 | 服务器内部错误 |

## 架构说明

### 后端三层架构

**Repository 层** - 数据访问层：
- 封装数据库操作，提供通用 CRUD 接口
- 各模型专用查询方法
- 事务管理支持

**Service 层** - 业务逻辑层：
- 实现核心业务逻辑
- 调用 Repository 进行数据访问
- 统一错误处理
- 业务规则验证

**Controller 层** - HTTP 控制层：
- 处理 HTTP 请求和响应
- 参数校验
- 调用 Service 层处理业务
- 返回统一响应格式

### 状态管理
- 前端使用集中式 Store 管理全局状态
- 用户认证状态、站点配置、通知状态统一管理
- 支持响应式状态更新

### 权限系统
- 基于权限组的细粒度权限控制
- system 组（级别 100）拥有所有权限
- admin 组（级别 80）管理普通用户内容
- 普通用户组按权限组配置访问权限
